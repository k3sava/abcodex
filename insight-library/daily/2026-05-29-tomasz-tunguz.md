Two posts from Tunguz this week, both pointing at the same shift. On May 27, he argued that software is no longer about UX or data -- it is about the harness, the coordination layer that turns an LLM into a callable, reliable agent. On May 26, he named what comes after data gravity:

> "If data gravity defined the last decade, agent gravity will define this one. As AI agents become the primary interface to enterprise data, the platforms running them will capture an outsized share of value -- and they're already locking the doors."

The pairing matters. The harness is the product; agent gravity explains the pricing power. Whoever controls the agent runtime controls the renewal. This is a packaging story as much as a product story. Clients who assume AI layers are interchangeable will discover switching costs embedded in the harness, not the model.

For consulting: this is the argument to use with clients who plan to swap AI vendors freely. The lock-in is structural, not contractual. For substrate: the harness layer -- tool registry, memory, routing, state persistence -- is where the architectural bet belongs. Build that first, not the model wrapper. For ab-codex: Tunguz's "agent gravity" framing is exactly the kind of named concept the library should capture with operator attribution before it becomes overused jargon.
